Project Overview

lean_c is a research and engineering project focused on defining Lean-level objects and relationships (theorems) that can be used to generate portable C code and to express machine-checkable proofs about code transformations and properties. The aim is to enable generation of small, predictable C suitable for resource-constrained environments while providing formal Lean constructions to reason about the produced code.

Source docs: README.md, doc/roadmap.md, doc/ideas.md

Key Benefits

  • Portable C output with minimal runtime dependencies.
  • Path toward machine-checked guarantees (safety, correctness).
  • Small binaries suitable for embedded and constrained targets.

Explicit Non‑Goals

  • Full Lean 4 → full C99 coverage.
  • Heavy OS runtime or GC dependency.
  • Replacing verified compilers like CompCert; interoperate conceptually.

Motivation

  • Lean 4 is excellent for writing verified programs and proofs.
  • However, produced binaries are large, making deployment in constrained or embedded environments difficult.
  • C remains the most widely supported systems language, especially where binary size, predictability, and platform support matter. Moreover, it is relatively easy to implement for new platforms compared to higher-level languages.
  • There is currently no practical, proof-aware pipeline from high level abstractions to low-level C code.
  • Bridging the two enables formally verified small, fast, and correct low-level software.

Project goals and scope

  • Define a Lean-level intermediate representation of a program module.
  • Provide implementation of basic data structures, such as containers, lists, maps, stacks, queues, and heaps.
  • Produce a small, provably correct library for operations on basic datatypes.

Technical Approach

  • Target a small, well-specified C subset, like Clight. This reduces the effort, still providing working programs. Combined with CompCert this will produce provably correct compiled executable for critical applications.
  • Define a Lean-level intermediate representation suitable for formal proofs
  • Proofs are broken into several categories:
    • Correctness of C syntax — allows defining a smaller subset of syntactic elements bound by usage rules.
    • Type preservation/transformation — modern programming requires clear typing; here we use strong typing.
    • Memory safety — prevents access to memory belonging to the system or other programs.
    • Resource bounds — guarantees about memory and other resource usage for a given algorithm.
    • Runtime bounds — complexity classes for execution time. Multithreading proofs aim for Rust-like guarantees while targeting simpler-to-implement C.
    • Absence of undefined behavior.
  • Once formal proofs of the above are provided generate readable, portable C code that can be compiled by standard toolchains, including CompCert.

Guarantees & Scope

  • Scope: small, well-defined C subset (e.g., Clight) with portable output.
  • Formal guarantees: syntax correctness, type preservation, memory safety, resource and runtime bounds, and absence of undefined behavior.
  • Non‑goals: full Lean 4 → full C99 coverage; heavy OS runtime or GC; replacing verified compilers (aim to interoperate with CompCert).

Current Status

  • Early-stage prototype; Lean modules under development (Types, Expr, Values, etc.).
  • Formal correctness proofs in progress; design docs available.

Roadmap & Milestones

Milestone 1

Define a Lean-level intermediate representation for type and value constructions
  • C types
  • Memory Block types, like Array, Slice.
  • Create a literal representations, like strings, numbers.
  • Provide a set of theorems/definitions about correctness of C lexems usage.

Milestone 2

Define a Lean-level intermediate representation for expression/statement semantics.
  • Introduce types suitable for theorem statements.
  • Theorems and Proofs: type preservation/transformation, memory safety, runtime bounds.

Milestone 3

Define a Lean-level intermediate representation for functions and modules
  • Theorems required for memory safety and resources used by functions.
  • Module generation in actual C.

Milestone 4

Implementation of basic data structures, like container, list, map, stack, queue, heap
  • Theorems proving implementation for basic data structures.
  • Generation and validation of C code for basic structures.
  • Performance/cost model for selected algorithms — design a complexity classes representation beyond simple ProgramTerminates.

Detailed roadmap: doc/roadmap.md
